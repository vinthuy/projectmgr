┌─────────────────────────────────────┐
│   ScreenDetail.vue (单一组件)       │
│                                     │
│   ├── 顶部操作栏                     │

• 模板：~250 行
• 脚本：~300 行
• 样式：~150 行

## 幻灯片 9：API 设计

```
🔌 RESTful API 设计

Screen 管理（5个端点）
GET    /screens              → 列表查询
GET    /screens/:id          → 详情查询
POST   /screens              → 创建
PUT    /screens/:id          → 更新
DELETE /screens/:id          → 删除

Tab 管理（3个端点）
POST   /screens/:id/tabs     → 添加 Tab
DELETE /screens/tabs/:id     → 删除 Tab
PUT    /screens/:id/tabs/reorder → 调整顺序

字段管理（3个端点）
POST   /screens/:id/items    → 添加字段
DELETE /screens/items/:id    → 移除字段
PUT    /screens/:id/items/reorder → 调整顺序

字段资源（1个端点）
GET    /field-definitions    → 获取可用字段

✅ 设计规范
• RESTful 风格
• 统一响应格式
• 完善的错误处理

### 常见问题预判

**Q1: 为什么不采用多组件架构？**
A: 当前功能集中在单页面，单一组件更简单，符合 KISS 原则。后续如需复用可再提取。

**Q2: 拖拽性能如何保证？**
A: 使用 vuedraggable（基于 Sortable.js），经过大量项目验证。当前数据量 < 50，性能优秀。大数据量时可启用虚拟滚动。

**Q3: 如果用户误删除怎么办？**
A: 危险操作有二次确认。P2 阶段会实现回收站功能，支持恢复。

**Q4: 是否支持移动端？**
A: 当前优先保证桌面端体验。P1 实现平板端，P2 实现移动端。响应式方案已设计好。

**Q5: 如何处理并发冲突？**
A: 采用乐观锁机制。检测到冲突时提示用户刷新页面，避免数据覆盖。
